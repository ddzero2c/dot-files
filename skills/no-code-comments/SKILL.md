# No Code Comments

## Rule

MUST NOT write code comments. Exception: the logic is truly obscure and cannot be expressed through naming or structure.

## The exception bar

A comment is justified only when ALL of these hold:

- Renaming the symbol cannot carry the meaning.
- Extracting a helper cannot describe the intent.
- Tightening the type / introducing a constant cannot encode the constraint.
- **If the comment were deleted, the failure mode would be silent.** A naive refactor or a future maintainer would get something subtly wrong, and *no* tooling — tests, types, linters, build, snapshot diffs — and *no* 30-second grep would surface it.

If any of the first three is "yes I could," refactor — do not write the comment. If the fourth is "tests/grep would catch it," the tooling already encodes the rule — do not write the comment.

## Common rationalization traps

These FEEL like WHY but are actually grep-redundant or tool-redundant:

- **"Field X is held for Y"** — `grep X\.` shows every caller. Drop.
- **"This calls `StripPrefix` because the spec uses bare paths"** — `StripPrefix("/api", ...)` is right there in the code; the spec is one open-file away. Drop.
- **"Snapshots / tests depend on this string / shape — keep it stable"** — change it, the test fails loudly. Tooling enforces. Drop.
- **"This file lives here / mounts on root mux because …"** — structural decisions read out of the wiring file directly. Drop.
- **"Used only by X" / "kept for Y to use"** — one grep. Drop.

The pattern: if a hypothetical confused reader could answer the question with a grep, a test run, a glance at the adjacent file, or a build error, the comment isn't carrying load — the tooling is. Comments are only for failures that escape all of those.

## Exemptions (do NOT delete)

Some "comments" aren't really inline comments — they're user-facing interface contracts that happen to live in source files because no better surface exists. Keep these:

- **CLI / executable script top-of-file usage docstring** — `#!/usr/bin/env bash` (or Python / Node / Deno shebang) followed immediately by a `# Usage: ...` block describing the script's purpose, subcommands / flags, I/O conventions, and any setup prerequisites. This is the script's `--help` analog, surfaced by `head script.sh`. Applies to any file run directly by humans (`*.sh`, `*.py` with shebang, single-file `node` / `deno` scripts, etc.) — even when the same info is duplicated in nearby SKILL.md / README.
- **License / copyright headers** required by org policy or external dependencies.
- **Generated-file banner** (`DO NOT EDIT — generated by X`) on codegen output.

Silence-test still applies *inside* these blocks — don't bloat a usage block with redundant prose, restated subcommand names, or marketing language. But don't strip the block entirely.

Inline comments inside script / function bodies still follow the no-comments rule normally.

## Always delete

- Explains *what* the code does — well-named identifiers already do that.
- Restates the function name or signature in prose.
- Names a caller ("Used by X", "Called from Y") — belongs in commit message / PR.
- "Added for X", "handles the Z flow" — rots as the codebase evolves.
- TODO / FIXME / WIP — handle now or open a ticket; do not ship the marker.
- Section banners inside a function ("// validate inputs", "// build query") — extract a helper instead.
- Doc comments that paraphrase the signature (`/// Returns the user by id.` on `getUserById`).
- Doc comments explaining a symbol's purpose, scope, or expanded use cases
  — that's the name failing to carry the meaning. Rename until the doc reads
  as redundant, then delete it. If no single name fits, the symbol is doing
  two jobs — split it.

## Pre-commit audit

Before `git commit`, scan the staged diff for added comment lines. For each, in order:

1. Can a rename / extracted helper / tighter type make it unnecessary? → refactor, drop the comment.
2. Does it match a row in "Common rationalization traps"? → delete.
3. **Silence test**: if you deleted this line, what would the next maintainer get wrong, and would *any* of {tests, types, linters, build, snapshot diff, one grep, reading the adjacent file} surface it? If yes → delete; the tooling carries the load. If no → keep.

One pass, not two. Don't grant "this explains a structural decision" or "this names a hidden invariant" as automatic passes — run them through the silence test.

Re-stage and commit. State briefly what was removed; do not ask permission per-comment.
